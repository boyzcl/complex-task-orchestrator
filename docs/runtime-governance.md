# Runtime Governance

最后更新：2026-04-17

## 目标

在不引入复杂自动化系统的前提下，让 runtime memory 从“会增长”变成“会克制增长”。

## 状态层级

### `active`

- 默认可读
- 当前仍被认为能稳定帮助决策
- 优先被回忆

### `watch`

- 允许存在
- 有一定价值，但证据仍偏少，或边界还不够稳定
- 只有在明确相关时才读取

### `stale`

- 默认不读
- 曾经有价值，但帮助感下降，或已被更稳规则替代
- 保留作为历史资产，不默认删除

## 最小治理规则

- 新 promote 的资产，默认先进入 `watch` 或在证据充分时进入 `active`
- 如果某条资产连续多轮没有带来帮助，或 recall 明显变脏，应考虑降为 `watch`
- 如果某条资产长期不再帮助判断，或已被更稳定规则取代，应标记为 `stale`
- `stale` 资产保留在仓库里，但不应进入默认读取集合

## 当前读取顺序

1. `active` pattern
2. `active` failure mode
3. 明确相关的 `watch` 资产
4. 默认跳过 `stale`

## 当前不做的事

- 不做自动 retire
- 不做自动 score
- 不做 embedding 检索
- 不做大规模资产扩张

## 评审触发点

当出现以下任一情况时，应回看治理状态：

- promoted 资产数量开始快速增长
- reuse 记录增加，但帮助感下降
- 同类任务频繁读到不相关资产
- 某类 failure mode 已被更强规则覆盖
