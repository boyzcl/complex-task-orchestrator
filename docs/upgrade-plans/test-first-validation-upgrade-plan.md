# complex-task-orchestrator 测试先行升级方案

日期：2026-04-29

状态：待用户确认

## 1. 文档目的

这份文档用于定义 `complex-task-orchestrator` 的下一次协议升级方案。

本次升级不直接改变 skill 本体，而是先把升级目标、流程改动、落盘产物、验证方式、实施步骤和自检结论写清楚。用户确认后，再进入正式实施。

## 2. 用户提出的两个核心要求

### 要求一：复杂或长链路任务必须先澄清到高置信

当任务复杂、长链路、需求不完整，或者 agent 对任务没有非常明确的理解时，agent 应主动向用户询问和确认更多信息细节。

进入正式执行前，agent 应达到约 95% 以上的需求理解把握，并完成明确的需求复述与用户确认。

### 要求二：复杂或长链路任务必须测试先行

复杂或长链路任务不应直接从执行计划开始。

正确顺序应是：

1. 先形成测试用例、验收标准和检验思路
2. 再生成完整执行计划
3. 将执行计划落到本地文档
4. 基于计划执行完整结果
5. 执行完成后，按预先设计的测试验证思路检查结果
6. 如果结果不符合预期，继续迭代补充和优化
7. 直到结果符合预期，或明确说明外部阻塞点

## 3. 当前 skill 满足情况

### 已基本满足的部分

当前 `SKILL.md` 已经比较充分地覆盖了要求一：

- 复杂、开放、多阶段、需求不完整任务会触发该 skill
- 默认一次只问一个关键问题
- 优先澄清目标、成功标准、约束、交付物和验证方式
- 当对用户真实目标有约 95% 把握时，完整复述需求
- 未确认前不要进入正式执行
- 如果执行中出现影响主路径的新歧义，需要回到澄清阶段

### 只部分满足的部分

当前 skill 对要求二只覆盖了后半段：

- 有自检
- 有质量标准
- 有完成后修订
- 有验证意识

但当前还没有把“测试先行”写成强制主流程。

具体缺口是：

- 没有强制在执行计划前先写测试用例或验收用例
- 没有默认落盘独立的验证方案
- 没有要求执行完成后逐项回验预设测试
- 没有把“不通过就迭代”写成正式循环
- `扩展验证` 当前偏向可选 forward test，不等同于主流程必需验证

## 4. 升级目标

本次升级目标是把当前流程从：

```text
需求澄清 -> 共识确认 -> 执行指令生成 -> 正式执行 -> 自检修订 -> 任务总结
```

升级为：

```text
需求澄清 -> 共识确认 -> 测试与验收设计 -> 执行计划生成 -> 正式执行 -> 预设验证回验 -> 失败迭代 -> 自检总结
```

升级后的 skill 应满足：

- 复杂或长链路任务在执行前必须先明确怎么验证结果是否合格
- 执行计划必须建立在测试与验收方案之后
- 本地文档必须记录验证方案和执行计划
- 完成后必须按预设验证方案逐项检查
- 未通过验证时必须迭代，不能只用主观自评分收尾
- 对简单、低风险任务仍保留轻量路径，避免过度流程化

## 5. 推荐主流程

### 阶段 1：需求澄清

目标：确认任务真实目标、范围、约束、交付物和成功标准。

规则：

- 一次只问一个真正影响方向的问题
- 优先问会改变执行路径的问题
- 如果用户回答仍模糊，继续追问
- 不在此阶段输出完整方案或开始执行
- 对复杂或长链路任务，不应仅靠假设继续

退出条件：

- agent 对需求理解达到约 95% 把握
- 已能清楚复述任务目标、范围、交付物、约束和验收标准

### 阶段 2：共识确认

目标：把需求理解变成用户确认过的执行边界。

必须覆盖：

- 当前任务是什么
- 目标是什么
- 不做什么
- 交付物是什么
- 关键约束是什么
- 初步验收标准是什么
- 是否允许写文件和运行测试

退出条件：

- 用户明确确认
- 或用户拒绝继续澄清时，agent 明确写出假设、风险和影响

### 阶段 3：测试与验收设计

目标：在执行计划之前，先定义结果如何被判断为合格。

这是本次升级新增的主阶段。

必须产出：

- 测试用例或验收用例
- 检验思路
- 验证方法
- 通过标准
- 证据留存方式

建议使用表格：

```md
| ID | 验证点 | 方法 | 预期结果 | 证据 | 是否必过 |
| --- | --- | --- | --- | --- | --- |
| V1 | 需求边界是否被覆盖 | 对照需求共识检查 | 所有必做项都有对应交付 | 文档引用 | 是 |
| V2 | 核心结果是否可运行/可用 | 运行测试或人工检查 | 无阻塞错误 | 命令输出/截图/文件 | 是 |
```

不同任务类型的验证方式应不同：

- 代码任务：单元测试、集成测试、构建、lint、类型检查、手工 smoke test
- 前端任务：浏览器截图、交互检查、响应式检查、控制台错误检查
- 文档任务：结构完整性、口径一致性、证据支持检查、读者路径检查
- 研究任务：证据等级、引用覆盖、反证边界、不确定性标注
- skill / workflow 任务：触发场景、路径选择、产物生成、失败路径、runtime reuse 边界
- 项目整改任务：文件存在、脚本通过、文档一致、验收矩阵更新

退出条件：

- 验证方案足以判断任务是否完成
- 每个核心交付物至少有一个对应验证点
- 必过项和可选项已区分

### 阶段 4：执行计划生成

目标：基于已确认需求和测试方案，生成可落地的执行计划。

执行计划必须晚于测试与验收设计。

必须包含：

- 执行步骤
- 涉及文件或模块
- 每步预期产物
- 每步对应的验证点
- 风险和回退策略
- 不做事项

执行计划应落盘为本地 Markdown 文档。

退出条件：

- 计划能直接指导执行
- 计划中的每个关键步骤都能对应到验证方案
- 用户未要求暂停或修改方向

### 阶段 5：正式执行

目标：按本地执行计划完成交付物。

规则：

- 执行时以计划为操作协议
- 如果出现影响目标、范围或验收标准的新歧义，暂停并回到共识确认
- 如果只是局部实现细节，可以做最小合理假设并记录
- 执行过程中发现计划不适用，应同步更新计划和原因

退出条件：

- 核心交付物已经完成
- 所有计划内必须动作已经处理

### 阶段 6：预设验证回验

目标：按阶段 3 设计的验证方案逐项检查结果。

必须执行：

- 逐项跑测试或人工核验
- 记录通过、失败、跳过和原因
- 保存关键证据，例如命令结果、文件路径、截图、对照表
- 不允许只用“看起来没问题”替代验证

建议结果表：

```md
| ID | 验证点 | 结果 | 证据 | 备注 |
| --- | --- | --- | --- | --- |
| V1 | 需求边界是否被覆盖 | Pass | 见 01/03 文档 | 无 |
| V2 | 自动化检查是否通过 | Fail | pytest 输出 | 需修复缺失 fixture |
```

退出条件：

- 所有必过项通过
- 或未通过项已明确属于外部阻塞，并说明影响

### 阶段 7：失败迭代

目标：验证不通过时继续补充、修复和优化。

规则：

- 任何必过项失败，都不能直接进入最终总结
- 必须先定位失败原因
- 修复后重新执行相关验证
- 如果连续迭代仍失败，要说明外部阻塞或风险

退出条件：

- 必过项全部通过
- 或明确说明无法继续的外部阻塞点

### 阶段 8：自检总结

目标：在测试验证之后，再进行最终主观质量自检。

自检必须回答：

- 需求是否真的被解决
- 预设验证是否通过
- 哪些地方仍有残余风险
- 是否有未完成的可选扩展验证
- 当前方案信心是多少

重要变化：

- 自检不能替代测试验证
- 自评分必须基于验证结果
- 如果必过验证失败，不能给出 9.5 以上评分

## 6. 推荐落盘产物

升级后，复杂或长链路任务默认最小文档集应调整为：

```text
docs/
  task-slug/
    01-requirement-consensus.md
    02-validation-and-test-plan.md
    03-execution-plan.md
    04-self-review-and-summary.md
```

### 01-requirement-consensus.md

记录最终确认版需求，不记录完整探索过程。

建议包含：

- 当前任务
- 目标
- 范围
- 非目标
- 交付物
- 约束
- 初步验收标准
- 用户确认状态

### 02-validation-and-test-plan.md

新增核心文档。

建议包含：

- 验证目标
- 测试/验收用例表
- 必过项与可选项
- 自动化测试命令
- 人工检查方法
- 证据留存方式
- 失败后的迭代规则

### 03-execution-plan.md

替代或吸收当前的一次性执行指令文档。

建议包含：

- 角色与执行原则
- 具体执行步骤
- 文件或模块范围
- 每步对应验证点
- 风险与回退
- 输出要求

为了兼容现有概念，可以在文档内保留一节 `一次性执行指令`，但文件职责应从“prompt”升级为“可执行计划”。

### 04-self-review-and-summary.md

记录验证结果和最终总结。

建议包含：

- 验证结果表
- 未通过项处理记录
- 自评分
- 扣分原因
- 实际修订
- 问题、本质原因、解决方案、信心
- 主任务是否完成
- 可选扩展验证是否仍存在

## 8. 具体协议改动建议

### 8.1 更新 Stage Broadcast

新增或调整阶段为：

```text
需求澄清
共识确认
测试与验收设计
执行计划生成
正式执行
预设验证回验
失败迭代
自检总结
扩展验证
```

其中 `预设验证回验` 是主流程，不应默认归入扩展验证。

### 8.2 更新 Quick Path

Quick Path 仍可保留，但必须加入最小验证设计。

快速路径的新规则：

- 可以压缩追问为 0 到 1 轮
- 仍要做简短共识确认
- 仍要列出最小验收检查
- 可以把验证计划和执行计划合并成短版
- 完成后仍要按最小验收检查回验
- 仍要保留自检总结

### 8.3 更新 Definition of Done

主任务完成条件应从：

- 需求已经确认
- 执行指令已经生成
- 核心交付物已经完成
- 已进行了自检
- 已给出最终总结

升级为：

- 需求已经确认
- 测试与验收方案已经生成
- 执行计划已经生成并落盘
- 核心交付物已经完成
- 已按预设验证方案回验
- 必过项已通过，或外部阻塞已说明
- 已进行了自检
- 已给出最终总结

### 8.4 更新 Self-Review

自检清单必须明确：

- 自检发生在预设验证之后
- 自评分必须引用验证结果
- 必过验证失败时不能给出高分
- 低于 9.5 时应优先回到失败验证项，而不是只润色总结

### 8.5 更新 Artifact Rules

默认文档骨架从 3 件套升级为 4 件套。

如果任务很轻，可以压缩为 2 件套：

```text
01-requirement-consensus.md
02-execution-and-validation-summary.md
```

但复杂或长链路任务不应省略验证设计。

本次升级完成后，应至少通过以下验证。

- artifact conventions 包含 4 件套
- definition of done 包含预设验证回验
- self-review checklist 明确自检不能替代验证

### V2：文档一致性验证

检查：

- README 没有继续只描述旧 3 件套流程
- first-run example 已展示测试先行路径
- validation matrix 已说明此次升级后的验证口径
- stage broadcast 阶段名称一致

### V3：复杂任务模拟验证

给一个需求不完整的长链路任务，观察 agent 是否：

- 先追问
- 不直接输出执行计划
- 达到高置信后才复述并请求确认
- 确认后先写验证方案
- 再写执行计划

### V4：清晰任务 Quick Path 验证

给一个目标清楚的小型复杂任务，观察 agent 是否：

- 不过度追问
- 仍列出最小验收检查
- 执行后按检查项回验

### V5：失败迭代验证

设计一个会导致首次测试失败的任务，观察 agent 是否：

- 记录失败项
- 修复问题
- 重新运行相关验证
- 通过后再总结

### V6：不适合自动测试的任务验证

给一个文档或研究类任务，观察 agent 是否能用人工验收表、证据对照表、口径一致性检查替代代码测试，而不是放弃验证。

## 11. 非目标

本次升级不做：

- 不把所有简单任务强制变成重流程
- 不要求每个任务都有自动化测试
- 不引入大型 benchmark 平台
- 不做复杂 memory 自动索引
- 不把可选 forward test 和主流程验证混为一谈
- 不在用户确认方案前修改 skill 本体

## 12. 风险与控制

### 风险一：流程变重

控制方式：

- 保留 Quick Path
- 允许简单任务合并验证和执行计划
- 只对复杂或长链路任务强制完整测试先行

### 风险二：测试形式化

控制方式：

- 要求每个验证点绑定交付物或需求条目
- 要求记录证据
- 要求失败后迭代

### 风险三：文档数量增加导致摩擦

控制方式：

- 默认 4 件套只用于复杂或长链路任务
- 轻量任务可以合并文档
- 后续可增加脚手架降低创建成本

### 风险四：非代码任务不知道如何测试

控制方式：

- 在 validation template 中提供任务类型映射
- 明确人工验收、证据矩阵、口径一致性检查都算验证

### 风险五：历史案例与新流程不一致

控制方式：

- README 明确 v0.2.0 是旧流程验证结果
- 新流程作为下一阶段升级
