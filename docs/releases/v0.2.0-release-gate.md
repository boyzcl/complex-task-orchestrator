# v0.2.0 发布门槛

日期：2026-04-17

## 0. 当前判断

按本文件定义的门槛回看当前状态：

> `complex-task-orchestrator` 已满足进入 `v0.2.0` 的最低通过条件。

本文件继续保留 gate 定义，供后续版本回看，而不是继续把项目描述为“尚未满足”。

## 1. 版本目标

`v0.2.0` 不应该代表“文档更全”，而应该代表：

> 这个项目已经从“有协议、有最小 runtime、有公开案例”推进到“真实任务中开始表现稳定，并出现最小可证明的 runtime reuse”。

如果这个条件没有成立，即使文档再多，也不应该进入 `v0.2.0`。

## 2. 当前项目所处阶段

项目在定义 gate 时更接近：

- 协议层已成形
- runtime 宿主已存在
- 公开证据层已建立
- 真实行为验证不足
- reuse 证据链尚未成立

而在 Round 1 完成后，项目已经进入：

`v0.2.0 ready`

## 3. Must-Pass 发布门槛

以下条件必须全部满足，才允许进入 `v0.2.0`。

### Gate 1：完成一轮真实任务验证

至少完成 6 个真实任务评分：

- 研究 / 综述类 2 个
- 规划 / 设计类 2 个
- 实现 / 整改类 2 个

判定标准：

- 每个任务都有评分卡
- 至少 4 个任务被判断为 `Clearly helpful`

### Gate 2：证明 quick path 不是摆设

必须至少有 1 个真实任务明确使用 `Quick Path`，并且评分结果支持：

- 这是正确选择
- 没有因为过度压缩导致明显跑偏

如果所有任务都只能走完整路径，说明 skill 的边界还不够成熟。

### Gate 3：形成最小 runtime reuse 证据链

必须完成至少 2 次真实 reuse 记录。

最小证据链要求：

1. 某个任务读取了已有 promoted asset
2. 记录读取原因
3. 记录读取是否带来帮助
4. 如果没有帮助，也要明确记录

如果 reuse 记录仍为 0，则不能进入 `v0.2.0`。

### Gate 4：确认至少 1 个新 failure mode

必须从真实任务中确认至少 1 条新的 failure mode，而不是只沿用旧案例里的问题。

原因：

- 如果运行中没有暴露新失败模式，通常意味着验证密度不够
- failure mode 是 agent 项目真正开始“用出来”的关键证据之一

### Gate 5：完成一轮 skill 本体修正

Round 1 结束后，必须至少基于真实任务结果修正一次以下任一内容：

- trigger boundary
- quick path / full path 选择条件
- failure handling

如果真实任务没有反推回 skill 本体，说明验证还没有真正进入闭环。

## 4. Should-Have 但非阻塞项

以下内容值得完成，但当前不是 `v0.2.0` 的硬阻塞条件。

### 4.1 新增更多公开案例

有帮助，但不是 release 的决定性证据。

### 4.2 更自动化的 runtime 管理

例如：

- 自动写 capture
- 自动更新 manifest
- 自动 triage

这些可以进入后续版本，但当前不应抢占行为验证主线。

### 4.3 更完整的客户端兼容验证矩阵

当前需要的是至少一轮真实任务验证，而不是立刻铺大矩阵。

## 5. 当前明确不是发布门槛的内容

以下内容即使做得更多，也不能单独构成 `v0.2.0` 充分条件。

- references 变多
- README 更长
- 案例写得更漂亮
- 目录结构更完整
- runtime 文件数量变多

这些都可能是进展，但不是版本升级证据。

## 6. Round 1 结束后的发布判断规则

Round 1 结束后，按以下逻辑判断：

### 允许进入 `v0.2.0`

当且仅当：

- 5 个 Must-Pass gate 全部满足

### 暂不进入 `v0.2.0`

只要出现以下任意一项：

- 真实任务不足 6 个
- `Clearly helpful` 少于 4 个
- quick path 没有真实通过案例
- reuse 记录少于 2 次
- 没有新 failure mode
- 验证结果没有反推 skill 本体修正

## 7. 推荐执行顺序

当前最合理的顺序是：

1. 补齐 3 个新真实任务
2. 在这些任务里做至少 2 次 runtime reuse
3. 汇总评分卡
4. 修正 `SKILL.md`
5. 再判断是否进入 `v0.2.0`

## 8. 当前结论

截至 2026-04-17：

- 当前项目已经满足 `v0.2.0`
- 当前最关键工作已从“补齐 Round 1”切换成“启动 Round 2 客户端验证”
- 只要客户端热路径行为验证仍不足，就不应把项目描述为更成熟的版本
