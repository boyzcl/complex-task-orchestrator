# First Run 示例

这份文档用来说明：一个陌生用户第一次调用 `complex-task-orchestrator` 时，通常会看到什么。

## 输入示例

```text
这是一个复杂任务。请用 complex-task-orchestrator 先问清楚需求，再生成执行指令，然后开始做。
```

## 默认路径

### 1. 澄清

如果任务仍有关键歧义，skill 会先问一个真正影响方向的问题，而不是立刻给完整方案。

### 2. 共识确认

当理解足够清楚后，skill 会复述：

- 任务是什么
- 目标是什么
- 约束是什么
- 交付物是什么
- 验收标准是什么

然后请求确认。

### 3. 执行指令生成

如果允许落盘，skill 会为当前任务生成一份一次性的执行指令文档，而不是套一个通用模板。

### 4. 正式执行

执行中如果出现新歧义：

- 影响主路径：回到澄清
- 不影响主路径：做最小合理假设继续

### 5. 自检与总结

完成后会：

- 打分
- 解释扣分原因
- 如果分数不足则修订
- 最终总结问题、本质原因、解决方案和信心

## Quick Path

如果目标、交付物和约束已经清楚，skill 可以压缩流程：

- 跳过多轮澄清
- 做一次简短共识确认
- 写一份短版执行指令
- 保留自检与总结

## 默认产物

在允许写文件时，推荐的最小产物是：

```text
docs/
  task-slug/
    01-requirement-consensus.md
    02-task-execution-prompt.md
    03-self-review-and-summary.md
```

如果不适合写文件，也可以保留口头版共识和压缩版执行指令。
