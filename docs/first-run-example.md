# First Run 示例

这份文档用来说明：一个陌生用户第一次调用 `complex-task-orchestrator` 时，通常会看到什么。

## 输入示例

```text
这是一个复杂任务。请用 complex-task-orchestrator 先问清楚需求，再先给测试/验收思路，生成执行计划，然后开始做。
```

## 默认路径

### 1. 澄清

如果任务仍有关键歧义，skill 会先问一个真正影响方向的问题，而不是立刻给完整方案。

### 2. 共识确认

当理解足够清楚后，skill 会复述：

- 任务是什么
- 目标是什么
- 约束是什么
- 交付物是什么
- 验收标准是什么

然后请求确认。

### 3. 测试与验收设计

如果允许落盘，skill 会先为当前任务生成测试/验收方案，说明结果要怎么被判断为合格。

### 4. 执行计划生成

在验证方案之后，skill 会生成一份一次性执行计划，而不是套一个通用模板。

### 5. 正式执行

执行中如果出现新歧义：

- 影响主路径：回到澄清
- 不影响主路径：做最小合理假设继续

### 6. 预设验证回验

完成交付物后，skill 会按预先设计的验证方案逐项检查。

如果必过项未通过，应先迭代修订并重新回验。

### 7. 自检与总结

完成后会：

- 记录预设验证结果
- 打分
- 解释扣分原因
- 如果分数不足则修订
- 最终总结问题、本质原因、解决方案和信心

## Quick Path

如果目标、交付物和约束已经清楚，skill 可以压缩流程：

- 跳过多轮澄清
- 做一次简短共识确认
- 列出最小验收检查
- 写一份短版执行计划
- 完成后按最小验收检查回验
- 保留自检与总结

## 默认产物

在允许写文件时，推荐的最小产物是：

```text
docs/
  task-slug/
    01-requirement-consensus.md
    02-validation-and-test-plan.md
    03-execution-plan.md
    04-self-review-and-summary.md
```

如果不适合写文件，也可以保留口头版共识、压缩版验收检查和执行计划。
