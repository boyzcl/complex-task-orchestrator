# Product Value Decision

日期：2026-04-17

## 决策

当前项目继续按：

**`Codex`-only 受限试用阶段**

来推进。

这次补跑后，决策没有变，但证据比上一版更强，试用范围也更清楚。

当前不进入的阶段仍然是：

- 泛化试用产品
- 跨客户端试用产品
- “所有复杂任务都更省事”的对外叙事

## 为什么这样决策

这次补跑后的信号已经足够清楚：

1. 短状态同步 / brief 已经出现重复正例  
   `T3`、`T5`、`T6` 都是 `Clearly helpful`，说明这不是单点偶然结果。

2. 快速 review / audit 已经成为最强价值场景  
   `T4`、`T7`、`T8` 都是 `Clearly helpful`，而且能直接降低文档口径漂移和证据表述偏满的返工风险。

3. 固定来源复杂判断可以试，但仍偏重  
   `T1` 和 `T9` 都支持“有价值，但更重”的判断，说明这类任务适合纳入试用范围，但不能讲成稳定、广谱的复杂代理能力。

4. 执行推进类任务仍然不该主打  
   `T2` 仍是 `Marginal`，说明“大而全执行推进包”现在还不适合回到主线。

所以当前最合理的动作不是扩大承诺，而是：

> 在已经重复成立的任务类型上继续做 `Codex` 内受限试用，同时把叙事和范围收紧到证据支持的边界。

## 试用范围

优先试用的任务类型：

- 状态同步稿 / brief
- 快速 review / audit
- 有固定来源清单的复杂判断 memo

其中第三类要额外满足一个前提：

- 必须显式锁定来源清单，避免把新生成工件继续卷回上下文

## 暂不主打的任务类型

- 执行推进包生成
- 需要长链交付且强依赖稳定收尾的任务
- 会不断吸入新工件、容易上下文自污染的开放复杂任务
- 任何需要被表述成跨客户端稳定成立的使用场景

## 接下来怎么做

### 1. 先把受限试用范围持续写实

后续对内或对外描述时，统一强调：

- 当前成立范围是 `Codex`
- 当前最强场景是 brief、quick audit、固定来源复杂判断
- 当前不是“通用复杂任务代理”

### 2. 如果继续补样，仍然围绕已成立范围扩

优先顺序不变：

- 状态同步 / brief
- 快速 review / audit
- 固定来源复杂判断

不要把主线重新拉回大而全执行包。

### 3. 如果再测执行推进类任务，先拆小粒度

只建议拆成更小单元再测，例如：

- 命令清单生成
- 回写步骤生成
- 命名约定生成

分别看清楚哪一段不稳，再决定要不要回到整包生成。

### 4. 只有同一问题重复出现时，才继续回写 `SKILL.md`

本轮补样没有出现新的重复协议问题，因此当前不需要把修改 `SKILL.md` 重新抬成主线。

## 这一步不做什么

- 不继续优先改 `SKILL.md` 主体
- 不把 `Claude Code` 补跑重新抬成当前主线
- 不扩大到“跨客户端已经成立”的叙事
- 不把两类复杂判断正样本误判成“广谱稳定”
- 不把入口文档里的宽表述当成当前证据已经支持
