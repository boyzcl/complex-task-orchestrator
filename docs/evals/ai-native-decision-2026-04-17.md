# AI Native Decision

日期：2026-04-17

## 1. 当前判断

从 AI native 视角看，这个项目的主瓶颈已经先后经历了两次切换：

1. 从“协议设计是否足够好”切到“客户端验证基础设施是否足够可靠”
2. 再从“runner 是否够稳”切到“跨客户端环境是否可达”

当前状态可以简化成：

- `protocol-native`：已成立
- `memory-native`：最小闭环已成立
- `eval-native`：按当前 `Codex` 验收口径已成立
- `runner-native`：已基本成立
- `cross-client-native`：仍未成立，但已不再是当前发布阻塞项

真正卡住下一阶段的，不是当前版本是否成立，而是后续是否还要继续追求跨客户端覆盖：

1. `Claude Code` 在当前机器上没有真实可运行入口
2. `Codex` 已能稳定拿到 Quick / Full Path 证据，并足以支撑当前版本结论
3. 项目后续若继续扩验证，缺的是“跨客户端可达性”，而不是继续改协议

## 2. 决策

后续主线不再继续优先修改协议，当前版本先按 `Codex` 验证通过收口：

> 以 `Codex` 客户端验证通过作为当前成立标准；`Claude Code` 补跑保留为增强项。

这意味着：

- 当前可以结束“是否成立”的争论
- 暂时不需要让 Claude Code 环境阻塞继续卡住结论
- 暂时不继续优先改 `SKILL.md`
- 如果后续要做增强，再补跨客户端验证

## 3. 为什么这是更 AI native 的路径

AI native 项目一旦从“设计阶段”进入“行为验证阶段”，最重要的就不再是文档更全，而是：

- 运行是否可复现
- 失败是否可留痕
- 摩擦是否可区分为“协议问题”还是“执行环境问题”

现在这个项目已经具备协议层、记忆层、最小稳定 runner，以及 `Codex` 侧真实热路径证据；这些已经满足当前版本成立所需的最小条件。

## 4. 本轮直接执行什么

本轮直接做三件事：

1. 把当前结论正式收口为“`Codex` 验证通过即可”
2. 把评估文档中的硬性跨客户端门槛降级
3. 保留 Claude Code 补跑信息，但不再阻塞当前版本判断

## 5. 完成后的意义

做完这一轮后，项目会从：

- “Codex 客户端内稳定成立”

推进到：

- “按当前验收口径正式成立”

这比继续被不可用客户端环境拖住更能推动下一阶段。

## 6. 下一步

当前这一步已经完成。基于最新结果，后续最合理的动作是：

1. 如果要做增强，再在有真实 Claude Code 客户端的环境里补跑 `A1 / A2`
2. 只在新增客户端也出现重复偏差时，再继续改协议
3. 当前阶段把重点放在版本收口，而不是继续追逐跨客户端完备性
