# Limited Trial Scope Memo

当前判断：接下来，`Codex`-only 受限试用应定义为“只覆盖短输出、高判断密度、输入边界清楚的任务”，重点放在状态同步、快速 review 和固定来源复杂判断，不把执行推进或跨客户端验证包装成主线。

## 可以主动试用的任务类型

现有样本里最稳的是短状态同步和快速 review / consistency check：目标清楚、输出边界清楚，几乎不需要补上下文，能直接减少整理和后续返工。另一个可以主动试用的是固定来源清单下的复杂判断 memo。它已经显示出真实价值，但前提是来源清楚、范围封闭、不要把新生成工件继续卷回上下文，否则过程会变重。

## 暂不应主打的任务类型

现阶段不应主打中等复杂执行推进、执行包生成、长链交付以及强依赖稳定收尾的任务。这类任务的问题不是不能理解，而是从“理解目标”到“稳定交付完成件”仍不稳。也不应主打会持续吸入新工件的开放复杂任务；同时，不要把 Claude Code 的结构兼容或后续补跑重新抬成当前试用叙事。

## 下一批样本怎么补最能提高判断质量

最有效的补样不是继续拉大场景，而是在已成立范围内扩密度：补 2 到 3 个状态同步 / brief、2 到 3 个快速 review / audit、1 到 2 个固定来源复杂判断，优先看它是否持续保持低补充上下文、低返工和不过度流程化。对执行推进类，只建议拆成更小单元测，再决定是否回到整包生成。

## 当前最容易说大的风险是什么

最容易被说大的，是把“有限但真实的产品价值”提前讲成“广谱复杂任务代理已经成立”。现有证据只支持 `Codex` 内的受限试用，不支持泛化承诺；如果把一次复杂任务复跑成功、结构兼容，或 `Codex` 内成立，外推成跨客户端和广谱稳定，后续判断会失真。
