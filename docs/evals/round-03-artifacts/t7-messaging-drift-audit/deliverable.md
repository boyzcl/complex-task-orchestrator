# T7 Messaging Drift Audit

日期：2026-04-17

检查范围：

- `README.md`
- `README_EN.md`
- `docs/validation-matrix.md`
- `docs/claude-code-compatibility.md`
- `docs/why-this-helps.md`

审计基准口径：

- 当前成立范围是 `Codex`-only 受限试用
- 当前最强场景是短状态同步、快速 review / audit、固定来源复杂判断
- 当前不应被表述成跨客户端稳定成立，也不应被表述成通用复杂任务代理

## 高价值发现

1. `README.md:3,7-12,32-47` 与 `README_EN.md:3,7-12,32-47` 仍然把产品主叙事放在“复杂、开放、多阶段任务”的通用编排上，还直接举了系统设计、项目规划、开放式实现这类大任务。这个表述会让外部读者默认把它理解成通用复杂任务代理，而不是当前已经相对成立的窄场景工具。再加上“clarify, align, execute, self-review”这套默认流程写得很完整，读者很容易把它读成稳定的广义 agent 工作流。
最小修正建议：把主定位改成 `Codex` 里的受限试用 workflow，首屏场景收窄到短状态同步、快速 review / audit、固定来源复杂判断，并把系统设计/开放式实现等例子移到“未来验证方向”或直接删掉。

2. `README.md:76-102,128-138` 与 `README_EN.md:71-97,123-133` 把 `Codex` 和 Claude Code 并列放进安装与当前状态，视觉上会被读成两个客户端都已进入可对外讲述的成立范围。即使后文另有边界说明，顶层 README 先给出对等安装入口，本身就会制造“跨客户端已基本成立”的第一印象。当前口径要求的是 `Codex`-only 受限试用，所以这里的结构已经先于文字边界把口径说宽了。
最小修正建议：在 README 首层明确写成“当前仅按 `Codex` 受限试用成立”，把 Claude 相关内容降级为实验性附录或单独边界说明，不再与 `Codex` 并列呈现为主安装路径。

3. `docs/claude-code-compatibility.md:5,11-26,79-86` 虽然写了“完整体验仍需验证”，但“结构兼容：已具备”“项目级入口：已具备”“安装脚本：是”这组结论仍然过于像一份正向兼容声明。对外或半对外读者通常不会细分“结构兼容”和“行为成立”，更容易把这页读成“Claude Code 基本可用，只差补充验证”。这会直接冲掉当前“不要表述成跨客户端稳定成立”的口径。
最小修正建议：把标题和结论改成更强边界版本，例如“Claude Code: 结构可见，但不在当前成立范围内”，并把判断语句统一改成“仅证明仓库结构与安装脚本存在，不构成当前客户端成立声明”。

4. `docs/validation-matrix.md:41-43,52-60,66-71` 与 `docs/why-this-helps.md:3,11-27,35-42` 叠加后，会把读者带到“这已经是一套对复杂任务普遍有效的机制”上。前者用了“6 个真实任务评分”“当前 6 个都为 Clearly helpful”“多任务 runtime reuse 效果”等证据型措辞，后者又用“模型”“复杂任务”这类泛化表达来讲价值，缺少 `Codex` 限定和强场景限定。这样的组合会把现有证据从“窄场景下的有限正结果”抬高成“通用复杂任务方法论已被验证”。
最小修正建议：在 validation matrix 里给所有效果性结论补上 `Codex` 与任务类型限定，在 why-this-helps 里把“复杂任务”改成当前已验证的三个强场景，并删弱对长期复用、广义交接等容易外推到通用代理的话术。

## 结论

这 5 个固定来源里，最高风险的漂移不是“明显说错”，而是通过主叙事、安装并列、兼容结论和泛化价值表达，把当前口径自然抬成了“跨客户端、面向广义复杂任务”的产品印象。若只做最小收口，优先顺序应是：先收 README 首屏定位，再收 Claude 兼容页标题与结论，最后给 validation / value 文档补上 `Codex` 与强场景限定。
