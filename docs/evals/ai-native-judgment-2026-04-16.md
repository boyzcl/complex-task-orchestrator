# AI Native 视角判断

日期：2026-04-16

## 1. 当前项目处于什么阶段

从 AI native 视角看，`complex-task-orchestrator` 已经不再是一个单纯的 prompt 包装项目，而是一个进入早期可用阶段的协议型 agent 项目。

但它还没有进入真正的“会越用越强”的阶段。

更准确地说，它现在处于：

- `协议已成立`
- `运行时宿主刚建立`
- `公开证据初步具备`
- `真实行为验证不足`
- `runtime reuse 证据链尚未成立`

## 2. 为什么说它已经不是普通 prompt 项目

当前项目已经具备几个明显的 AI native 特征：

### 2.1 它在定义默认行为，而不只是输出文本

`SKILL.md` 现在定义了：

- 触发边界
- quick path 和完整路径
- 阶段播报
- 失败处理
- runtime reuse 规则

这意味着它不是单次回答模板，而是在定义 agent 的默认动作协议。

### 2.2 它已经开始拥有最小 runtime 层

当前项目已有：

- `runtime/captures/`
- `runtime/promoted/patterns/`
- `runtime/promoted/failure-modes/`
- `runtime/state/runtime-memory-manifest.json`

这说明它已经不再只是 repo-native 项目，而开始具备 runtime-native 的雏形。

### 2.3 它已经开始区分资产层

项目已明确区分：

- 展示资产
- curated case
- runtime 资产
- 安装副本

这非常重要，因为 AI native 项目最容易出现的错觉，就是把“写出来的规则”误当成“真实运行中的记忆”。

## 3. 为什么说它还没有进入真正的 AI native 飞轮

目前最关键的缺口不是文档，而是行为与复用。

### 3.1 真实行为验证仍然不足

当前通过的是：

- 结构验证
- 安装脚本验证
- 文件与区块存在性验证

当前还缺：

- Codex 客户端里的真实触发验证
- Claude Code 客户端里的真实触发验证
- quick path / full path 选择是否稳定
- 自检是否真的带来行为修正

### 3.2 runtime 还没有形成可证明的复用链

目前项目已有：

- capture
- promoted pattern
- promoted failure mode

但仍缺最关键的一步：

> 某次任务产出经验 -> 经验被提升 -> 下一次任务明确读取 -> 下一次任务结果因此变得更稳

如果没有这个证据链，runtime 目前仍然更像“整理过的记忆草案”，而不是“已成立的 agent 记忆机制”。

### 3.3 项目还没有形成自增强增长飞轮

真正 AI native 的项目，会出现这种结构：

1. 真实任务跑起来
2. 运行时留下高质量痕迹
3. 痕迹被筛选为 pattern / failure mode
4. 下一轮读取这些资产
5. 下一轮表现更稳
6. 又产生更高质量的 runtime 资产

当前项目还停在第 2 到第 3 步之间。

## 4. 当前最重要的战略判断

当前主线不应该继续放在：

- 扩写 references
- 美化 README
- 增加更多理论解释
- 提前做复杂自动化 memory 系统

当前主线应该明确切换到：

- `真实任务行为验证`
- `runtime reuse 证据链建立`

换句话说：

这个项目现在最需要的不是“写得更完整”，而是“跑得更真实”。

## 5. 决策

后续只做一条主线：

> 用一轮真实任务验证，把项目从 document-native 推进到 runtime-native。

## 6. 决策理由

### 理由 1：协议层已经够用了

当前 `SKILL.md` 和 supporting docs 已经足以支撑下一阶段验证。

再继续扩协议，边际收益会明显下降。

### 理由 2：现在最大的真实不确定性在行为层

现在不确定的不是“别人能不能看懂”，而是：

- 是否能稳定触发
- 是否能稳定选对路径
- 是否能真的减少返工
- 是否能让 runtime 资产对下一轮有帮助

### 理由 3：只有真实任务才能决定后续该优化哪里

如果不先跑一轮真实任务，你无法知道后续该：

- 继续扩 runtime
- 收紧 skill 边界
- 调整 quick path 条件
- 重写 failure handling

## 7. 后续应该做什么

后续阶段按下面顺序推进：

1. 建立 `Round 1` 真实任务验证框架
2. 连续跑 6 个真实任务
3. 为每个任务记录触发、路径选择、结果收益、capture / promote / reuse 情况
4. 基于结果修正 skill 本体
5. 再决定是否进入下一阶段 release

## 8. 本轮已执行的动作

这次不是只输出判断，已经同步启动以下执行基础设施：

- 建立 `docs/evals/`
- 建立 `Round 1` 执行计划
- 建立任务评分卡模板
- 建立 runtime reuse 日志
- 建立任务板
- 用现有三个案例生成基线评分卡

也就是说，项目已经从“知道下一步该做什么”，进入“下一步已经开工”的状态。
