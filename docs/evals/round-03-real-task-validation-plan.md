# Round 3 Real Task Validation Plan

日期：2026-04-17

## 1. 目标

Round 3 不再优先验证“能不能触发”，而是验证这套 skill 在真实工作里是否真的省事。

本轮重点只看四件事：

- 是否明显减少手动补充上下文的次数
- 是否减少返工
- 是否更稳定地产出可执行下一步
- 是否会在短任务上显得流程正确但过重

## 2. 当前冻结规则

从本轮开始，先冻结协议版本：

- 不继续改 `SKILL.md` 主体
- 只允许修小 bug 和明显歧义
- Round 3 的主要动作放在真实任务验证，而不是继续改协议本体

## 3. 验证范围

本轮使用 `Codex` 客户端完成 4 个真实任务位：

- `T1` 高歧义复杂决策任务
- `T2` 中等复杂、需要推进执行的任务
- `T3` 短平快状态同步任务
- `T4` 短评审 / 一致性检查任务

这些任务都应尽量贴近当前项目真实会做的工作，而不是纯评测题。

## 4. 每个任务必须记录什么

- 原始用户任务
- 任务类型
- skill 是否触发
- 路径选择是否合理
- 是否减少了额外解释成本
- 最终结果是否可直接拿来用
- 是否给出了可执行下一步
- 是否出现“任务不大但流程过重”
- 是否存在值得推广的 runtime reuse

## 5. 执行方式

本轮统一使用 `run-codex-client-validation.sh`：

- 继续隔离 `HOME/.codex`
- 保留 `events.jsonl`、`last.txt`、`status.json`
- 允许 `workspace-write`，但输出限定在各自 artifact 目录

## 6. 通过判断

Round 3 可以判定为“有产品化价值”，当且仅当：

- 3 到 5 个真实任务里，大多数被判定为 `Clearly helpful` 或 `Helpful but heavy`
- 至少 2 个任务明确减少了手动补充上下文
- 至少 2 个任务明确减少了返工风险
- 至少 1 个短任务暴露了“流程过重”或“没有过重”这类真实边界
