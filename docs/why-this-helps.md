# 为什么这类 skill 会有帮助

`complex-task-orchestrator` 想解决的不是“模型不会做”，而是“模型太容易在需求还不清楚时就开始做”。

## 当前已被真实样本支持的帮助范围

截至 2026 年 4 月 17 日，当前最有证据支持的帮助范围不是“所有复杂任务”，而是三类场景：

- 短状态同步 / brief
- 快速 review / audit
- 有固定来源清单的复杂判断

这些场景的共同点是：

- 目标清楚
- 输出边界清楚
- 返工风险高于多花一点前置判断成本

## 直接提问的常见问题

- 任务目标模糊，但模型会自动补全成一个看似完整的答案
- 用户以为双方在说同一件事，实际上理解层已经分叉
- 输出结束后没有预设验证和真正的自检，返工被推迟到下一轮

## 这个 skill 的核心价值

### 1. 把共识前置

不是先写答案，而是先确认问题到底是什么。

### 2. 把验证标准前置

不是做完以后才问“好不好”，而是先定义测试用例、验收标准和检验思路。

### 3. 把执行协议显性化

不是凭感觉临场发挥，而是先为当前任务写一份一次性执行计划。

### 4. 把返工前移

通过共识确认、验证先行和回验迭代，把“晚发现的问题”尽量提前发现。

### 5. 把复杂任务切成用户能感知的阶段

阶段播报可以减少用户在复杂流程中的不确定感。

## 它不是什么

- 不是万能 prompt
- 不是所有任务都该套的重流程
- 不是自动替代判断力的框架

## 最适合它的场景

当任务满足下面任意几点时，它通常会比直接提问更稳：

- 可能存在多种完全不同的理解路径
- 一旦走偏，返工成本很高
- 输出需要交接、留档或长期复用
- 结果是否合格需要先定义测试或验收方式
- 用户自己也还没完全想清楚任务边界

## 当前不该说成什么

- 不该把它说成所有复杂任务都默认更优
- 不该把一次复杂判断成功复跑说成广谱稳定
- 不该把 Claude Code 结构兼容说成跨客户端成立
