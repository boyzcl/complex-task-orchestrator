# 任务执行指令

这是一份针对“创建 `complex-task-orchestrator` Skill”这一任务的一次性执行指令。

```md
# 创建 `complex-task-orchestrator` Skill

## 角色

你是一名资深的 Skill 设计师与工作流架构师。你的任务是把一套已经验证过价值的复杂任务协作模式，封装成一个其他 Agent 也能稳定复用的 Codex Skill。

## 核心任务

在当前项目中创建一个名为 `complex-task-orchestrator` 的可复用 Skill，将其安装到本地 Codex Skill 环境，并让项目处于“可等待后续真实复杂任务做 forward test”的状态。

## 上下文

- 当前任务本身就在按照复杂任务流程执行
- 需求共识已经达成并已记录
- 项目中已经存在元指令文档 `元生成指令.md`
- 这个 Skill 不能只服务于“创建 Skill”场景，而必须泛化到其他复杂任务

## 交付物

1. `skill/complex-task-orchestrator/` 下的可复用 Skill 目录
2. 一份具备清晰触发规则和完整流程的 `SKILL.md`
3. 让主 Skill 保持精炼的支撑性 reference 文档
4. 位于 `docs/` 中的项目文档，包括需求共识、执行指令、自检总结
5. 安装到本地 Codex Skill 目录中的 Skill
6. 结构校验结果与最终自检总结

## 执行原则

- 从“最强大脑”视角出发，先定义真正的问题，再组织解决方案
- 保持用户偏好的完整流程：澄清 -> 确认 -> 生成执行指令 -> 执行 -> 自检 -> 总结
- 让 Skill 既可复用、又精炼、还能被触发
- 优先保证中文用户可读性，同时兼顾未来开源传播
- 默认采用严格流程，但在任务已足够清晰时允许适度压缩

## 实施步骤

1. 在 `docs/` 中创建本次任务需要的项目文档
2. 初始化并完善 `skill/complex-task-orchestrator/` 下的 Skill 结构
3. 编写真正可用的 `SKILL.md`，至少定义：
   - 什么时候应该触发该 Skill
   - 什么时候不应该触发
   - 默认的澄清与执行协议
   - 执行中出现新歧义时如何处理
   - 完成后如何自检与总结
4. 增补精炼的 reference 文档，覆盖：
   - 文档产物约定
   - 一次性任务执行指令生成
   - 自检与评分
5. 更新 `agents/openai.yaml` 中面向 UI 的元数据
6. 将 Skill 安装到本地 Codex Skill 目录
7. 运行结构校验
8. 记录自检结果；如果质量未达标，继续修订

## 输出要求

- Skill 主体要简洁、流程化、可执行
- 可复用规则放在 Skill 中，项目特定内容放在 `docs/`
- 执行指令文档要足够自包含，使另一个高能力模型也能直接接手
- 避免给可复用 Skill 塞入无必要文件

## 质量标准

- Skill 对原作者和陌生开源用户都应该可理解、可使用
- 触发描述要覆盖显式调用、意图触发和排除条件
- 工作流不能只停留在规划层，必须能指导真实执行
- 项目要准备好，等待后续另一个复杂任务来做真实 forward test

## 最终自检

执行完成后：

- 给当前工作打 1 到 10 分
- 说明扣分原因
- 如低于目标分数则继续修订
- 最终明确说明：
  - 原始问题是什么
  - 本质原因是什么
  - 已实施的解决方案是什么
  - 当前方案在实践中生效的信心有多高
```
