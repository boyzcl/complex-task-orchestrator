# 我做了一个专门处理复杂任务的 Skill

事情是这样的。

我前段时间在用 Codex 干活的时候，越来越强烈地感受到一件事。

很多任务，不是模型不够强。

而是任务一上来就错了。

错在哪儿呢。

错在我们总以为，复杂任务的难点在执行。其实很多时候，真正的坑，早在第一句需求里就已经埋下去了。目标没说清，边界没说清，交付物没说清，什么叫做做完也没说清。结果就是模型看起来已经开始干活了，文件也在生成，字数也在增加，甚至语气还挺笃定，但你心里就是隐隐约约知道，完了，这玩意大概率会返工。

这种感觉，用过 Agent 的朋友应该都懂。

尤其是那种开放性任务。写方案，做调研，梳理工作流，搭一个 Skill，做一个跨文件改造，或者查一个你自己都还没想明白的问题。你会发现模型越强，越容易在错误方向上跑得飞快。因为它真的太会补全了，太会把一个模糊的需求，补成一个看起来完整的答案。

这事儿最吓人的地方就在这。

它不是不会做。

它是太会做了。

于是我就在想，有没有可能，不是再去发明一条更长的 Prompt，也不是每次都手动提醒模型先别急，而是把这件事直接做成一个 Skill。一个专门用来处理复杂任务起手式的 Skill。

先别做。

先想清楚。

先问清楚。

先达成共识。

然后再做。

这个念头一出来，我就决定亲自下场试试。

然后比较骚的事来了。

我拿来验证这个 Skill 的第一个任务，就是创建这个 Skill 自己。

也就是说，我不是先拍脑袋写一堆规则，再说它大概能用。我是真的按它想要倡导的流程，从零开始走了一遍。先参考最强大脑的处理方式，再一轮一轮追问，再做需求确认，再生成一次性执行指令，再正式执行，再自检。就这么硬走了一遍。

你别说，这个过程比我预想中有意思很多。

因为我很快发现，这个 Skill 要解决的问题，根本不是帮模型写出更漂亮的提示词。

它要解决的，是复杂任务里那个最常被忽略的断层。

用户脑子里想的是一件事。

模型理解成的是另一件事。

最后产出的又是第三件事。

中间每一层，都觉得自己没问题。

这才是最麻烦的。

所以我后来给这个 Skill 定的名字，叫 `complex-task-orchestrator`。它不是一个某个垂直场景的小插件，不是专门写文章的，也不是专门做调研的。它更像一个复杂任务的编排器。它做的第一件事，不是回答你，而是逼着你和模型一起把任务讲清楚。

这个 Skill 现在的主流程，大概是这样的。

当任务本身复杂、开放、多阶段，或者需求明显不完整的时候，它不会直接开工。它会先进入需求澄清阶段，一次只问一个真正推进理解的问题。问到它觉得已经差不多有九成五把握了，它会停下来，不再继续装懂，而是完整复述一遍它理解的需求是什么，目标是什么，约束是什么，交付物是什么，然后等你确认。

这一步我现在是越来越信了。

因为很多返工，真不是能力问题，就是共识问题。

有了确认之后，它不会立刻乱冲。它会再生成一份只属于当前任务的一次性执行指令文档。注意，是当前任务，不是那种万能模板。也就是说，你这次要做医学综述，它就该写一份医学综述的执行协议。你这次要做 Skill，它就该写一份创建 Skill 的执行协议。每个任务都应该有自己的作战地图，而不是拿一份空泛的总纲到处套。

等这份文档有了，才进入正式执行。

执行完也还没结束。

后面还有自检。

这一块也是我特别想固化下来的。因为大模型很擅长交卷，不太擅长复盘。很多时候它做完了就收手了，但真正靠谱的工作，通常都要有一个回头看一眼的动作。所以我把这个动作强行写进了流程里。做完之后自己打分，说明扣分原因，如果分数不够高，就继续修，或者至少把外部阻塞点讲清楚。

这里面最让我兴奋的一点，不是它有多复杂，而是它终于把很多原来只存在于我脑子里的协作习惯，变成了一个可以复用、可以验证、可以传给别人的东西。

以前那种感觉很像什么呢。

很像你自己有一套做事的手感，但这套手感只能靠默契。你自己知道什么时候该停，什么时候该追问，什么时候该确认，什么时候该重来。可一旦换个模型，或者换个人协作，这些手感就散了。

现在不一样了。

现在这些东西被写进流程里了。

而且更有意思的是，这个 Skill 不是做出来就完了。我在做完第一版之后，真的拿它去跑了第二个完全不同方向的任务。是一个医学综述任务，问题是性行为相关的能力和欲望，到底受什么因素影响。这个任务本身也挺复杂的，因为它天然容易混淆欲望、能力、关系、满意度、疾病、药物、心理状态这些层层叠叠的变量。

如果直接开查，特别容易一头扎进资料堆里。

但这次我先让它走流程。先澄清范围，再确认是不是要给建议，再确认是不是面向广泛人群，再确认是不是优先高等级证据。等这些共识有了，再去做循证检索和综述。最后落出来的结果，明显比那种直接搜一圈就开始写的方式稳很多。

这一下我基本就确认了，这个东西不是一次性的。

它是真的有泛化能力。

当然，我也不是想把它吹得多神。

坦率的讲，第一版跑完以后，问题还是暴露出来了不少。最明显的一个问题，是用户文档语言。当时我和模型明明全程在用中文聊，结果生成出来的项目文档却是英文的。我看到这里的时候，一时间无语凝噎。因为这就很典型，你明明在追求开源友好，最后却把当前用户的可读性牺牲掉了。

这事后来我给改了。

我直接把规则写死，运行时生成的项目文档，默认跟随当前对话语言。你现在跟我用中文协作，那需求共识、执行指令、自检总结这些文件，就应该优先是中文。开源传播是开源传播，用户当前能不能看懂，是另一件更要紧的事。

第二个暴露出来的问题，是阶段感不够强。

说白了，不对，这词不能说。反正就是，我后来发现如果不明确告诉用户现在在哪个阶段，用户很容易懵。你会看到文件在生成，看到事情在推进，但你并不知道这是还在澄清，还是已经开始执行，还是主任务其实已经结束，后面只是在做额外验证。

所以我又补了一层阶段播报。

需求澄清，共识确认，执行指令生成，正式执行，自检修订，任务总结，扩展验证。每切一个阶段，都得让用户知道。主任务做完了，也得明说主任务已经完成。后面的 forward test，如果要做，那就是扩展验证，不要再说得像主流程还没收尾一样。

这个小改动，我觉得特别值。

因为它不是在优化能力，而是在优化协作体感。

很多时候，用户的不安，不是因为你做得不好，而是因为他不知道你现在到底在干什么。

第三个我后来补进去的，是轻量模式和完整模式的选择规则。

不是每个任务都值得上重流程。

这也是真的。

如果只是改个小 bug，或者整理一段短文，还要先来一轮单问澄清，再生成需求共识，再写执行协议，那就有点过于一本正经了。所以我后来给它补了一个决策逻辑。目标清不清楚，交付物清不清楚，约束清不清楚，验收标准清不清楚，现在直接做会不会大概率跑偏。如果这些问题里有好几个都答不上来，那就别装轻松，老老实实走完整流程。

如果都已经很清楚了，那就轻一点。

这就比较像开车。

不是所有路段都需要一档慢慢蹭，但你到了山路，到了夜路，到了大雾天，你还非要一脚油门，那就是找事儿。

还有一个我自己挺满意的升级，是我把研究型任务也单独沉淀成了一个 workflow。因为我发现调研、综述、证据整合这类任务，虽然也属于复杂任务，但它们有自己很鲜明的特点。先定义问题，再定证据优先级，再找高等级证据，再说明哪些结论证据强，哪些还不确定。这个节奏，和普通执行型任务还是不太一样。

所以到这里，这个 Skill 终于开始有一点我最想要的样子了。

它不再只是一个 Prompt。

它更像一个协作协议。

更具体一点，它像什么呢。

它像一个很靠谱的项目搭子。你把一个糊成一团的复杂问题甩给他，他不会立刻装作胸有成竹。他会先停一下，问你几句关键问题，把你脑子里的雾稍微拨开一点。等确认差不多了，他会把这件事重新给你讲一遍，确认咱们说的是同一件事。然后他再去写作战地图，再开工。做完也不急着邀功，还会回头看一眼，确认这东西到底有没有真的解决问题。

我自己现在越来越觉得，这种能力在 Agent 时代会特别重要。

因为模型会写，已经不稀缺了。

模型会查，也越来越不稀缺了。

真正稀缺的，是它能不能在复杂问题上，先和你形成真实的共识，再持续稳定地把任务往前推。

这个差别，看着不大。

但真的用起来，是两个世界。

一个世界里，你一直在补锅。

另一个世界里，你终于感觉有人在和你一起做事。

如果你也经常遇到这种情况，脑子里明明知道自己要做的是个大活，但又说不清从哪句话开始容易跑偏，那我觉得这个 Skill 可能会对你有帮助。它不保证每次都完美，也不是什么银弹。我自己都还在继续打磨它。

但至少，它已经把一件我以前总靠手感完成的事，变成了一个可以复用、可以验证、可以交给别人继续接力的东西。

我觉得这就挺重要的。

很重要。
