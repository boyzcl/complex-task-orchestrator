# Failure Handling

当复杂任务在执行中出现异常路径时，使用本参考。

## 1. 用户不愿多轮澄清

- 说明继续直接执行的风险
- 把关键假设明确写出来
- 在可接受风险内继续推进

## 2. 用户回答仍然模糊

- 继续追问当前最影响方向的一个问题
- 不要一次抛出多个分支问题

## 3. 不适合写文件

- 保留口头版需求共识
- 保留压缩版验收检查和执行计划
- 不要因为无法落盘就失去协议核心

## 4. 中途需求变更

- 如果改变目标、范围、交付物或验收标准，回到共识确认
- 如果只影响局部细节，记录变更并继续

## 5. 需要快速交付

- 走 quick path
- 保留最小共识确认
- 保留最小验收检查
- 完成后按最小验收检查回验
- 保留自检

## 6. 预设验证失败

- 不要直接进入最终总结
- 记录失败项和证据
- 先定位原因并修复
- 修复后重新运行相关验证
- 如果无法通过，说明外部阻塞、影响范围和剩余风险
