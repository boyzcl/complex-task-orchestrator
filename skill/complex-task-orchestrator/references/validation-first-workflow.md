# Validation First Workflow

用本参考处理复杂、长链路、返工成本高，或用户明确要求先设计测试/验收再执行的任务。

## 核心原则

- 先确认需求，再设计验证。
- 先定义合格结果，再写执行计划。
- 执行完成后，按预设验证方案回验。
- 必过项失败时，先迭代修正，再进入最终总结。
- 自检不能替代测试或验收验证。

## 适用场景

- 代码实现、项目整改、跨文件修改
- 前端页面、交互工具、游戏或可视化
- 文档体系治理、口径校准、审计整改
- 研究综述、证据整合、复杂判断 memo
- skill、workflow、agent 协议设计或升级

## 标准顺序

1. 需求澄清
2. 共识确认
3. 测试与验收设计
4. 执行计划生成
5. 正式执行
6. 预设验证回验
7. 失败迭代
8. 自检总结

## 测试与验收设计

在写执行计划前，先定义：

- 验证点
- 验证方法
- 预期结果
- 必过项与可选项
- 证据留存方式
- 失败后的迭代规则

每个核心交付物至少要有一个对应验证点。

## 执行计划生成

执行计划必须基于已确认的需求和验证方案。

计划中应写明：

- 每步要做什么
- 涉及哪些文件或模块
- 每步产出什么
- 对应哪个验证点
- 发现计划不适用时如何处理

## 预设验证回验

执行完成后必须回到验证方案逐项检查。

允许的验证方式包括：

- 自动化测试
- 构建、lint、类型检查
- 浏览器截图与交互检查
- 人工验收表
- 证据矩阵
- 口径一致性检查
- 读者路径检查

不允许只用“看起来没问题”替代验证。

## 失败迭代

如果必过项失败：

1. 记录失败项
2. 定位原因
3. 修复或补充
4. 重新运行相关验证
5. 通过后再总结

如果无法修复，必须说明外部阻塞和影响范围。

## Quick Path 下的最小版本

当任务目标、交付物和约束都清楚时，可以压缩流程，但仍应保留：

- 简短共识确认
- 最小验收检查
- 简短执行计划
- 完成后的验收回验
- 最终自检总结
